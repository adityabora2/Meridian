## 1. What The Project Is
 
A document Q&A system that decides *how* to retrieve before it retrieves — deployed serverlessly on AWS — grounded in three 2024 research papers.
 
**The core problem with standard RAG:**
Every question goes through the same fixed pipeline regardless of complexity:
```
Question → Search → Retrieve Chunks → Generate Answer
```
- Simple questions get over-retrieved — noisy context, worse answers
- Complex multi-hop questions get under-retrieved — one search isn't enough, hallucinations follow
 
**The solution:** Route by complexity first. Three modes, decided before touching the vector store.
 
---
 
## 2. Research Grounding
 
| Paper | Core Idea Implemented |
|---|---|
| Adaptive-RAG (Jeong et al., NAACL 2024) | Three-mode routing — classify query complexity before retrieval |
| Self-RAG (Asai et al., ICLR 2024) | Model reflects on whether retrieval is needed, critiques its own output |
| Chain-of-Verification (Dhuliawala et al., ACL 2024) | Verify every claim in the answer against retrieved evidence before returning |
 
This is not a tutorial project. It is a direct implementation of peer-reviewed research in a production AWS deployment.

## 3. The Three Modes
 
**Mode 1 — No Retrieval**
- Question: simple, LLM already knows the answer
- Example: "What is RAG?"
- Action: Claude Sonnet answers directly, zero vector search
- Why: cheapest, fastest, often more accurate than retrieving
 
**Mode 2 — Single-Hop Retrieval**
- Question: needs document grounding, one search is enough
- Example: "How does Self-RAG decide when to retrieve?"
- Action: one pgvector search → top-k chunks → Claude Sonnet generates with citations
- Why: standard retrieval, clean and fast
 
**Mode 3 — Multi-Hop Retrieval + Self-Critique Loop**
- Question: complex, needs evidence from multiple documents
- Example: "What retrieval strategies do Self-RAG and Adaptive-RAG share?"
- Action: Claude Sonnet decomposes into sub-questions → searches for each → chains evidence → generates answer → Claude Haiku checks every claim → loops back if unsupported → exits when fully grounded
- Why: only way to correctly answer cross-document reasoning questions

## 4. LangGraph — How It Works In This Project
 
LangGraph is the orchestration layer. It manages which node runs next, when to loop, and when to exit. It does NOT do retrieval or LLM calls — those happen inside each node.
 
**The graph:**
```
Question in
      ↓
ROUTER NODE (Claude Haiku)
→ classifies: easy / medium / hard
      ↓
┌─────────────┬──────────────┐
↓             ↓              ↓
EASY         MEDIUM         HARD
↓             ↓              ↓
DIRECT       SEARCH         DECOMPOSE NODE
ANSWER       NODE           (Claude Sonnet splits
NODE         ↓               into sub-questions)
↓            GENERATE        ↓
EXIT         NODE            SEARCH NODE (×n)
             ↓               ↓
             EXIT            GENERATE NODE
                             ↓
                        SELF-CRITIQUE NODE
                        (Claude Haiku checks claims)
                             ↓
                    ┌────────┴────────┐
                 CLEAN           UNSUPPORTED
                    ↓            CLAIMS FOUND
                   EXIT          ↓
                            loop back to
                            SEARCH NODE
                            (max 3 iterations)
```
 
**Three LangGraph concepts to know:**
- **Node** — a Python function that does one job (route, search, generate, critique)
- **Edge** — connects one node to the next
- **Conditional edge** — reads current state, decides which node runs next (enables the self-critique loop)
 
**LangGraph vs LangChain:** LangGraph sits on top of LangChain's core abstractions but is fundamentally different. LangChain is sequential chains. LangGraph is stateful graph-based orchestration with loops and conditional branching. You use LangGraph directly — LangChain comes in only as a dependency via `langchain-aws` for the Bedrock wrapper.

## 5. Simplified AWS Stack (Final Decision)
 
**5 services only. Everything else was cut.**
 
| Service | Role | Cost |
|---|---|---|
| **AWS S3** | Stores PDFs, triggers ingestion Lambda on upload | Free tier |
| **AWS Lambda** | Runs ingestion pipeline + query app (FastAPI) | Free tier |
| **AWS Bedrock** | Claude Haiku (routing + critique), Claude Sonnet (generation), Titan Embeddings | ~$5–10 total |
| **pgvector on RDS** | Vector store — PostgreSQL + pgvector extension | Free tier (db.t3.micro, 12 months) |
| **AWS CDK (Python)** | Provisions entire stack with one command | Free |
 
**Cut services and why:**
- ~~API Gateway~~ → replaced by Lambda Function URL (one line in CDK, no extra service)
- ~~Step Functions~~ → replaced by LangGraph (does the same stateful orchestration in Python code)
- ~~CloudWatch custom dashboards~~ → basic Lambda logs are automatic, skip custom dashboards for MVP
- ~~X-Ray~~ → nice to have, not needed for MVP
- ~~Mangum~~ → only needed with API Gateway, dropped both together
 
**Lambda Function URL (replaces API Gateway):**
```python
fn.add_function_url(
    auth_type=lambda_.FunctionUrlAuthType.NONE
)
```
Gives you a direct HTTPS endpoint. No extra service needed.

## 6. Full System Architecture
 
### Ingestion Pipeline (runs once per PDF upload)
```
PDF uploaded to S3
      ↓
S3 event triggers Lambda (ingestion)
      ↓
PyMuPDF — parses text, preserves layout
Unstructured.io — detects tables, strips headers/footers
      ↓
Chunks tagged with metadata:
{
  chunk_id, page_number, chunk_type (paragraph/table/figure),
  section_heading, document_name
}
      ↓
Titan Embeddings (via Bedrock) — generates vector per chunk
      ↓
pgvector on RDS — stores vector + metadata
```
 
### Query Pipeline (runs on every question)
```
User asks question via Streamlit
      ↓
FastAPI on Lambda (via Lambda Function URL)
      ↓
LangGraph graph starts
      ↓
Router Node → Claude Haiku classifies complexity
      ↓
Mode 1 → Direct answer → return
Mode 2 → pgvector search → Claude Sonnet generates → return
Mode 3 → Decompose → search (×n) → generate → self-critique → loop or return
      ↓
Answer returned with citations and mode label
```

## 8. MVP Dataset (Final Decision)
 
**Public dataset — the research papers themselves.**
 
Use 15–20 publicly available PDFs: Adaptive-RAG, Self-RAG, Chain-of-Verification, and related RAG/LLM reasoning papers from ArXiv.
 
**Why this dataset:**
- 100% public, zero confidentiality concern
- Domain directly motivates the architecture — you're building an adaptive RAG system over the papers that inspired it
- Test set of 30 questions writes itself because you've read these papers
- Demo questions are immediately legible to any AI engineer interviewer
 
**30-question test set structure:**
- 10 easy questions → should trigger Mode 1
- 10 medium questions → should trigger Mode 2
- 10 hard cross-paper questions → should trigger Mode 3
 
Build this test set in Week 1 alongside AWS setup. Do not leave it to Week 5.

The LangGraph graph never changes across versions. Only the ingestion pipeline and LLM provider get extended.

## 10. PII Masking (Version 2 Feature — Not MVP)
 
For the record — how it works when added:
 
spaCy NER runs locally before any chunk reaches AWS:
```
John Smith → [PERSON_1]
14/03/1985 → [DATE_1]
Apollo Hospital → [ORG_1]
₹50,00,000 → [MONEY_1]
```
 
Token mapping stored locally. AWS only ever sees masked text.
After answer generation, local de-masking swaps tokens back.
CloudWatch logs never contain real PII.

## 15. Known Roadblocks
 
| Roadblock | Reality | Mitigation |
|---|---|---|
| AWS IAM errors | Most common week 1 pain point. `AccessDeniedException` with no obvious cause. | Use IAM policy simulator. Budget 10 days not 7 for week 1. |
| Bedrock model availability | Not all Claude models available in ap-south-1 (Mumbai) | Use us-east-1. ~200ms extra latency, no architectural change. |
| LangGraph conditional edges | Loop doesn't converge or exits prematurely. Debugging is non-obvious. | Cap iterations at 3. If loop breaks, ship Mode 3 without self-critique. |
| Lambda cold starts | LangGraph import overhead = 3–5 second cold start. Looks broken in demo. | Add provisioned concurrency to query Lambda in CDK. |
| Motivation drop after week 2 | Infrastructure is done but nothing intelligent is happening yet. | Get Mode 2 working by end of week 2. First grounded answer fixes motivation. |

## 18. Key Decisions Log (What Changed From Original Plan)
 
| Decision | Original | Final | Reason |
|---|---|---|---|
| Orchestration | AWS Step Functions | LangGraph | More relevant for AI engineering roles, same stateful loop capability |
| Vector store | OpenSearch Serverless | pgvector on RDS | OpenSearch costs $0.48/hr minimum. RDS free tier = $0 |
| API layer | API Gateway + Mangum | Lambda Function URL | Fewer services, same result |
| Dataset | ArXiv papers (generic) | The research papers themselves | Domain matches architecture, test set writes itself |
| Observability | CloudWatch custom dashboards | Basic Lambda logs | MVP simplification, add dashboards in version 2 |
| PII masking | MVP feature | Version 2 feature | Keeps MVP scope clean, still in the architecture story |
| Mode 3 | Required | Target, not required | Scope cut based on LLM Council timeline analysis |
